Thinking like a hacker:

Security considerations for
High-Fidelity Web Archives

Ilya Kreymer, Webrecorder

Jack Cushman, Perma.cc

Why is security a concern?

  • Web Archives just collection of old pages
  • High-Fidelity web archives run untrusted web software
  • Live site is "safe", so nothing to worry about
  • Web archive replay can pose new security risks

Possible security threats

  • Unauthorized capture
  • Manipulating capture or replay
  • Threats to users data
  • Threats to credibility of a web archive

Threat: Unauthorized capture

  • Capture system may have priveleged access to local network
  • Possible to capture local private resources, into a public archive

Mitigation: Network Filtering

  • Don't allow capture of local ip ranges
  • Run capture system in isolated sandbox

Concern: URL rewriting

  • Untrusted content accessed from web archive domain
  • Separate domains:
    • http://example-domain-A.com/
    • http://example-domain-B.com/
  • Rewritten replay, same domain:
    • http://example.com/http://example-domain-A.com/
    • http://example.com/http://example-domain-B.com/
  • Standard cross-domain protections do not apply!

Threat: Cross-Domain Cookie Access

  • Cookies of rewritten page may be accessible to other rewritten pages

Mitigation: Rewriting

  • Rewrite cookies to exact path only

Concern: Domain Cookies

  • Domain wildcard *.example.com cookies?
  • Client-side created cookies"

Threat: Local Storage Access

  • All local/session storage bucketed by host
  • Different sites can access, collide on storage keys

Mitigation: Additional Bucketing

  • Add host to storage key to bucket by host

Concern: Storage Overflow

  • Space for local/session storage is limited, per host.
  • One site may use up all the storage space still

Threat: Cookie Overflow

  • One page can wipe all cookies on a domain
  • Keep setting new cookies via document.cookie until old ones are dropped.

No Mitigation

  • Not possible to prevent cookie overflow!

Concern: web archive running on subdomain

  • http://example.com/ is the main institution home page
  • http://archive.example.com/ is a web archive
  • An archived page could pose a threat to the parent domain
  • document.domain can be set to parent domain

Threat: Session hijacking/manipulation

  • http://archive.example.com/replay/evil.html changes origin to parent domain
  • Archived page can now access http://example.com/
  • If user logged in, evil archived page can act as logged in user to http://example.com/
  • Can manipulate session, overflow cookies, etc...

Mitigation: Run web archive on separate domain

Mitigation: Run archive in containerized/proxy mode browser

Recommendation: Keep Web Archive Content Isolated

  • Use iframes for replay/capture of content
  • Load web archive app from "app domain"
  • Load iframe content from "content domain"
  • Webrecorder example:
    • https://webrecorder.io/ -- app domain
    • https://wbrc.io/ -- content domain
  • Perma.cc example:
    • https://perma.cc/ -- app domain
    • https://perma-archives.org/ -- content domain

Threat: Live Web Leakage

  • Accessing live web when page presented as archival undermines credibility
  • Malicious page can access live web to mislead, load alternative content

Mitigation: Content-Security-Policy header can limit access to web archive domain

Mitigation: Run archive in containerized/proxy mode browser

Threat: Anti-archiving anti-rewriting measures

  • URL rewriting replay is sophisticated but not foolproof
  • Determined advesary could detect if page is running in url rewriting mode
  • Impossible to detect every challenge

Mitigation: Don't use rewriting mode for capture/replay

  • Use containerized/proxy mode browser for capture/replay
